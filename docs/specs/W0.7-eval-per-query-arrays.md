# W0.7 — Enrich `eval_embeddings` return with per-query arrays

**Status:** spec — pending review
**Owner:** TBD
**Estimated effort:** 0.5 weeks
**Workstream dependencies:** W0 (uses `jammi-numerics` types)
**Workstreams blocked by this:** W4 (sample-based rules need per-query arrays at gate time)

## Motivation

The current `EvalRunner::eval_embeddings` collects per-query metrics in a `Vec<QueryMetrics>` (verified at `crates/jammi-ai/src/eval/runner.rs:97-101`), aggregates them to a mean via `RetrievalMetrics::aggregate`, then serializes ONLY the aggregate to `serde_json::Value` (line 107) and returns it. The per-query data is **computed and immediately discarded**.

W4's sample-based rules (`WelchTTestRule`, `MannWhitneyURule`) need two arrays of per-query metric values to run a statistical comparison between two model versions at gate time. Without per-query arrays, these rules degrade to a `NeedsHuman("no samples available")` verdict — band-aid behaviour that fails CLAUDE.md.

The fix is small: change the return type to expose both aggregate AND per-query arrays. Per CLAUDE.md "no backwards compatibility / reshape the function," do not add a parallel `_detailed` variant; change the return shape everywhere.

## Current state (verified at spec time)

- `crates/jammi-ai/src/eval/runner.rs:28` — `pub async fn eval_embeddings(&self, source_id, embedding_table, golden_source, k) -> Result<serde_json::Value>`
- `crates/jammi-ai/src/eval/runner.rs:97-101` — `query_metrics.push(RetrievalMetrics::compute_query(...))` builds the `Vec<QueryMetrics>` per query
- `crates/jammi-ai/src/eval/runner.rs:106-107` — `let agg = RetrievalMetrics::aggregate(&query_metrics); let metrics_json = serde_json::to_value(&agg)?;` — aggregate computed, per-query Vec dropped
- `crates/jammi-ai/src/eval/runner.rs:108-122` — eval run record persisted with `metrics_json = to_string(&agg)` (aggregate only)
- `crates/jammi-ai/src/session.rs:642` — public `pub async fn eval_embeddings(...)` re-exports the runner method with the same `serde_json::Value` return type
- Similar pattern for `eval_inference` (runner.rs:128) and `eval_compare` (runner.rs:211)

## Change

Introduce a typed return struct exposing both aggregate and per-query data. After W0 lands, `QueryMetrics` and `AggregateMetrics` live in `jammi_numerics::retrieval`.

```rust
// crates/jammi-ai/src/eval/mod.rs (or eval/report.rs, new)

use jammi_numerics::retrieval::{AggregateMetrics, QueryMetrics};
use serde::{Deserialize, Serialize};

/// Result of one `eval_embeddings` invocation. Exposes both the aggregate
/// over all queries (the historical return shape) and the per-query arrays
/// (required for sample-based statistical rules at gate time).
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct EmbeddingEvalReport {
    /// Mean metrics across all queries.
    pub aggregate: AggregateMetrics,
    /// One record per query, in golden-set order.
    pub per_query: Vec<PerQueryRecord>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PerQueryRecord {
    /// `query_id` from the golden source (the join key).
    pub query_id: String,
    pub metrics: QueryMetrics,
}
```

`EvalRunner::eval_embeddings` and `InferenceSession::eval_embeddings` change their return types from `Result<serde_json::Value>` to `Result<EmbeddingEvalReport>`. The runner builds `per_query` from the existing `query_metrics` Vec (no new computation; just retains the data it was already throwing away).

Similar change for the other two methods:
- `eval_inference` returns `InferenceEvalReport { aggregate: ClassificationResult | NerMetrics, per_record: Vec<PerRecordPrediction> }`
- `eval_compare` returns `CompareEvalReport { per_table: Vec<TableEvalReport> }` where each `TableEvalReport` is `{ table_name, embedding_eval }`

## Out of scope

- Persisting per-query arrays in the catalog `eval_runs` table — defer until a use case justifies the storage cost (the aggregate is what the catalog needs for historical reporting; per-query is a transient response shape)
- Adding new aggregate functions (median, percentile) — defer
- Streaming the per-query records — current behaviour is batch, return-all-at-once; that's fine for current workload sizes (typical golden set ≤ 10K queries)
